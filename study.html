分野別ポイント

試験で狙われるポイントと落とし穴

  • Aurora Global DB は RPO <1秒・RTO <1分。リージョン跨ぎのDR要件には第一候補。
  • RDS Multi-AZ は同一リージョン内のみ。リージョン障害には対応不可。Read Replicaは非同期なのでRPO=0にならない。
  • スパイク対策の3層: CloudFront(エッジキャッシュ) → ElastiCache(DBキャッシュ) → Auto Scaling(EC2追加)。Auto Scalingだけでは起動3〜5分かかる。
  • DR戦略の選択基準: RTO/RPO要件 と コスト許容範囲で決まる。「コスト効率」が問われたら Backup&Restore or Pilot Light。
  • Multi-Site Active/Active は最高コスト。「最もコスト効率が高い」問題では選ばない。
  • Step Functions: Standard(最大1年・非同期) vs Express(最大5分・高スループット)。試験では用途の使い分けを問われる。
  • IAMポリシー評価順: 明示的Deny → Allow → 暗黙的Deny。SCPのDenyはIAMのAllowを上書きする。
  • SCP+IAMの落とし穴: SCPでDenyしたうえでIAMでAllowしても効果なし。OU分離で例外アカウントを作る。
  • CloudHSM は FIPS 140-2 Level 3 専有ハードウェア。KMSカスタムキーストアと連携すれば既存アプリ変更なし。
  • 予防的vs検出的コントロール: SCP=予防的(発生させない)。Config Rules/GuardDuty=検出的(発生後に検知)。
  • VPCエンドポイント: Gateway型(S3・DynamoDB)は無料。Interface型は有料。インターネット/NAT不要でAWSサービスに安全接続。
  • Macie は S3内のPIIを機械学習で検出。GDPR/HIPAA対応の問題で登場する。
  • 購入オプション使い分け: 24h稼働→Reserved。中断可バッチ→Spot(最大90%引)。柔軟性重視→Savings Plans。BYOL→Dedicated Host。
  • Dedicated Host は BYOL専用。コスト削減目的では選ばない。物理サーバー占有でライセンス持込み専用。
  • S3 Lifecycleポリシー: アクセスパターンが既知ならLifecycle。不明ならIntelligent-Tiering。Glacierは取得遅延を許容できる場合。
  • Savings Plans の3種: Compute(最大66%・最も柔軟) / EC2 Instance(最大72%・特定ファミリー固定) / SageMaker。
  • NATゲートウェイのコスト: データ転送料が発生。VPCエンドポイント(Gateway)でS3/DynamoDBへのNAT経由を排除できる。
  • Transit Gateway は多数VPCのハブ&スポーク接続。VPCピアリングフルメッシュはn*(n-1)/2の接続数が必要で推移的ルーティング不可。
  • DX Gateway は1本のDirect Connect接続から最大10リージョンのVPCに接続可。Transit Gatewayと組み合わせると数百VPCまでスケール。
  • CloudFront vs Global Accelerator: HTTPアプリはCloudFront。非HTTP(ゲーム/VoIP)や静的IPが必要な場合はGlobal Accelerator。
  • Route 53ルーティング: フェイルオーバー / 加重(A/Bテスト) / レイテンシー / 地理的 / 近接性(Bias) / 複数値応答 の6種類。
  • PrivateLink はVPCをまたいでサービスをプライベート公開。VPCピアリングと違い推移的ルーティングが不要。
  • DMS CDC(継続レプリケーション) が最小ダウンタイム移行の基本。ソースDBを止めずに差分同期し、カットオーバー時のダウンタイムを数分に圧縮。
  • mysqldump は本番では非現実的。数TB規模では数時間〜数日のダウンタイム。最小ダウンタイム要件の問題では選ばない。
  • 移行ステップ: Application Discovery Service(現状把握) → Migration Hub(進捗管理) → DMS/SMS(実移行)。まず Discovery が先。
  • Snowball Edge は帯域が極端に低い場合の物理転送。ネットワーク帯域が十分あればDMS/DataSyncの方が効率的。
  • EMR は既存Hadoop/Spark/Hiveの移植に最適。GlueはサーバーレスSparkだがHadoopエコシステム全体の移植には制約がある。
アーキテクチャ図解

試験頻出パターンを視覚化

マルチリージョン DR — Aurora Global DB

障害時: Route 53 ARCがDNSを切替 → セカンダリAuroraがWriterに昇格(RTO <1分)

リアルタイムデータ収集パイプライン

Kinesis DS から Lambda(異常検知) と Firehose(S3蓄積) に並列配信。Athena で S3 をアドホッククエリ。

ハイブリッド接続 — DX Gateway + Transit Gateway

1本のDirect Connect → DX Gateway → Transit Gatewayで数百VPCを集約管理。VPCピアリングフルメッシュ不要。
